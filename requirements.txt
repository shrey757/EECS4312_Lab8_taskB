## 1. Problem Understanding

System purpose:
The system manages registration for a single event with a fixed capacity. It supports direct registration until capacity is reached, places additional users into a FIFO waitlist, automatically promotes the earliest waitlisted user when a registered user cancels, prevents duplicate registrations, and allows users to query their current status.

Inputs to the system:
- Event capacity as a non-negative integer
- User ID as a non-empty string
- Operation type: register, cancel, or status query

Outputs produced by the system:
- Updated registered list
- Updated waitlist
- User status result
- Clear outcome for accepted, rejected, or edge-case operations

Primary goal of the system:
To provide a predictable, consistent, and persona-aware event registration system that supports both efficient use for Jane and clear, trustworthy behavior for Mo.

---

## 2. Assumptions

A1: The system manages exactly one event at a time.
A2: Capacity is fixed during execution unless explicitly changed by a future requirement.
A3: User IDs are non-empty strings and uniquely identify users.
A4: Operations are processed one at a time in the order received.
A5: The waitlist must preserve FIFO ordering.
A6: The same valid operation sequence should always produce the same result.
A7: A user cannot exist in both registered and waitlisted states at the same time.
A8: Rejected or invalid operations must return an explicit outcome and must not silently change the system state.

---

## 3. Functional Requirements

FR1. Capacity Initialization  
The system shall accept a fixed event capacity that defines the maximum number of registered users.

FR2. Capacity Validation  
The system shall reject invalid capacity values less than 0 and return a clear outcome indicating invalid input.

FR3. Registration Until Capacity  
The system shall register users until the number of registered users reaches the defined capacity.

FR4. Waitlist Placement  
When capacity is full, the system shall place additional users into a FIFO waitlist.

FR5. Duplicate Prevention Across the System  
The system shall ensure that a user appears at most once anywhere in the system and cannot exist simultaneously in the registered list and the waitlist.

FR6. Duplicate Registration Handling  
If a user who is already registered or already waitlisted attempts to register again, the system shall reject the request as a duplicate and return a clear outcome.

FR7. Cancellation Support  
The system shall allow a user to cancel their entry whether they are registered or waitlisted.

FR8. Waitlist Order Preservation  
When a waitlisted user cancels their entry, the system shall remove that user and preserve the relative order of the remaining waitlisted users.

FR9. Automatic Promotion  
When a registered user cancels and at least one user exists on the waitlist, the system shall automatically promote the earliest waitlisted user to the registered list.

FR10. Status Query  
The system shall allow users to query their current status in the system.

FR11. Allowed Status Values  
The system shall return status results using a fixed set of states:
- registered
- waitlisted
- not_found

FR12. Waitlist Position Reporting  
When a user is waitlisted and queries status, the system shall return the user’s current waitlist position.

FR13. State Integrity for Rejected Operations  
Rejected operations, including duplicate registration attempts or cancellation of a non-existent user, shall not modify the registered list, waitlist, or capacity.

FR14. Capacity Zero Behavior  
If capacity equals 0, the system shall place all registration attempts on the waitlist and shall not register any users.

FR15. Explanation Support  
The system shall provide a short, understandable explanation when an action is rejected, when a cancellation changes the system state, or when a waitlisted user is promoted.

---

## 4. System Constraints

C1. Deterministic FIFO Ordering  
The system must preserve strict FIFO ordering in the waitlist so that promotions always occur in the same order for the same sequence of operations.

C2. Automatic Promotion Rule  
When a registered user cancels and the waitlist is not empty, the system must immediately promote the earliest waitlisted user without requiring manual action.

C3. Concise and Consistent Status Output  
The system must return status outputs using a fixed and limited set of states so that results are concise and easy to understand.

C4. Single Outcome per Operation  
The system must produce exactly one outcome for each operation request and must not generate duplicate confirmations or repeated outputs.

C5. Deterministic State Transitions  
For the same initial state and the same sequence of operations, the system must always produce the same registered list, waitlist order, and user statuses.

C6. Sequential Operation Processing  
Operations must be processed one at a time in arrival order so that ties or conflicting requests are resolved deterministically.

C7. Explicit Edge Case Handling  
The system must explicitly handle edge cases such as capacity = 0, cancellation of a non-existent user, repeated cancellations, duplicate registrations, and waitlist cancellations before promotion, and must return a clear outcome rather than failing silently.

C8. Clear Explanations for State Changes  
When an action is rejected, when a cancellation changes system state, or when a waitlisted user is promoted, the system must provide a short, understandable explanation.

## 5. Acceptance Criteria

AC1 – FIFO Promotion Order  
Given capacity 2 with u1 and u2 registered and u3 and u4 waitlisted in that order,  
When u1 cancels,  
Then u3 is promoted before u4, and the remaining waitlist order is preserved.

AC2 – Automatic Promotion  
Given capacity 1 with u1 registered and u2 waitlisted,  
When u1 cancels,  
Then u2 is automatically promoted to registered.

AC3 – Registered Status Query  
Given u1 is registered,  
When u1 queries status,  
Then the system returns registered.

AC4 – Duplicate Registration Rejection  
Given u1 is already registered,  
When u1 attempts to register again,  
Then the system rejects the request as a duplicate and returns one clear outcome.

AC5 – Deterministic Final State  
Given two identical valid operation sequences from the same initial state,  
When both sequences are executed,  
Then the same final registered list, waitlist order, and statuses are produced.

AC6 – Capacity Zero Behavior  
Given capacity 0,  
When u1 registers,  
Then u1 is not registered and is placed on the waitlist, and the system returns a clear outcome.

AC7 – Cancellation of Non-Existent User  
Given u9 is not in the system,  
When cancellation is requested for u9,  
Then the system returns an explicit not-found outcome.

AC8 – Sequential Cancellations with Promotion  
Given u1 and u2 are registered and u3 and u4 are waitlisted,  
When u1 and u2 cancel in sequence,  
Then u3 and u4 are promoted in FIFO order and the result is deterministic.

AC9 – Promotion Explanation  
Given a registered user cancels while another user is waitlisted,  
When promotion occurs,  
Then the system returns an explanation that the cancellation created an opening and the earliest waitlisted user was promoted.

AC10 – Duplicate Waitlist Registration  
Given capacity 1 with u1 registered and u2 waitlisted,  
When u2 attempts to register again,  
Then the system rejects the request as a duplicate and the system state remains unchanged.

AC11 – Rejected Operations Preserve State  
Given any valid current state,  
When a duplicate registration or non-existent cancellation is requested,  
Then the system returns a clear rejection and the registered list and waitlist remain unchanged.

AC12 – Non-Existent Status Query  
Given u9 is not present in the system,  
When u9 queries status,  
Then the system returns not_found.

AC13 – Invalid Capacity Input  
Given an initialization request with capacity -1,  
When the system validates the input,  
Then the system rejects the capacity as invalid and returns a clear outcome.

AC14 – Waitlist Cancellation Preserves Order  
Given u1 and u2 are registered and u3, u4, and u5 are waitlisted in that order,  
When u4 cancels their waitlist entry,  
Then the waitlist becomes u3, u5 and the relative order of remaining users is preserved.

## 6. Edge Cases

EC1: Capacity = 0  
Expected system behavior: If capacity is 0, all registration attempts must go directly to the waitlist and the system must return a clear outcome.

EC2: Cancel Non-Existent User  
Expected system behavior: If a cancellation is requested for a user not found in the system, the system must return an explicit not-found outcome instead of failing silently.

EC3: Multiple Cancellations in Sequence  
Expected system behavior: If multiple cancellations occur in sequence, the system must promote users in deterministic FIFO order after each cancellation.

EC4: Duplicate Registration  
Expected system behavior: If a user attempts to register twice, whether already registered or already waitlisted, the system must reject the duplicate request with one clear outcome only.

EC5: Waitlisted User Cancels Before Promotion  
Expected system behavior: If a waitlisted user cancels before being promoted, the system must remove that user without disturbing the order of the remaining waitlist.

EC6: Invalid Capacity  
Expected system behavior: If initialization is attempted with capacity less than 0, the system must reject the input and return a clear invalid-capacity outcome.

EC7: Non-Existent Status Query  
Expected system behavior: If a user who is not present queries status, the system must return not_found.

EC8: Rejected Operations Preserve State  
Expected system behavior: If a duplicate registration or non-existent cancellation is attempted, the registered list, waitlist, and capacity must remain unchanged.

---

## 7. Tie-Breaking Rules

TBR1:
Scenario: A registered user cancels while multiple users are on the waitlist.
Tie Condition: More than one waitlisted user is eligible to fill the newly available slot.
Decision Rule: The system must promote the user who entered the waitlist earliest, following strict FIFO order.
Example: If u3 joined the waitlist before u4, and a registered user cancels, the system promotes u3 before u4.

TBR2:
Scenario: Two identical valid operation sequences are executed from the same starting state.
Tie Condition: Multiple outcomes appear possible because the same conditions are repeated.
Decision Rule: The system must produce the same registered list, waitlist order, and status outputs every time for identical inputs and operation order.
Example: If one run registers u1, u2, u3 and then cancels u1, and another run performs the exact same sequence, both runs must end with the same final state.
